# AAAAT v1 release-readiness context seed

This context captures the accepted state after the PR37-based rebuild was merged back into `didacll/local-desktop-dashboard` as the new working basis. The accepted merge commit is `2b02f8de62543721cb247e7eedf19f0e4a2ac3ca`, produced from PR #43 head `e4575a61a3afc63d9514dd8951a8d830551d5100`. The historical PR37 state at `97b2e474d4c609002a4c786f81c60446e3b0be5e` remains the authority floor for desktop behavior.

AAAAT is a local-first, provider-agnostic, human-operated job-application workspace. Its v1 purpose is not a generic CRM, a web dashboard, or an autonomous job bot. It helps a user manage candidatures from raw offers through analysis, editable preparation, generated application materials, local artifacts, and sent-state tracking. The human remains the operator; agent work is bounded, local, and explicit.

The accepted v1 runtime direction is wx desktop. Earlier browser/server/FastAPI/dashboard work was removed from the active runtime surface. PR40 and PR41 are treated as failed/superseded evidence, not architecture. PR42 remains useful evidence only where it preserved PR37 projection/navigation contracts and added bounded local-runner, non-blocking wx execution, missing-keyword follow-up, offer-first intake, and supplementary CV/cover-letter action concepts.

The current accepted branch already contains the broad wx-only cleanup from the PR37 rebuild. Server/dashboard modules were removed. Project dependencies were reduced; wxPython remains the desktop optional dependency. CLI/docs were realigned around local storage, `aaaat-desktop`, and the local demo seed. The desktop demo seed now uses active v1 candidature fields and active/closed status semantics. SQLite schema work includes active/closed normalization, broader candidature-detail storage, artifact events, profile facts, task runner groundwork, and todo schema compatibility for pinned/due fields.

Several launch/runtime failures from the rebuild have already been fixed. The projection serializer no longer dereferences legacy `module.view` or `module.region` attributes. Smart View no longer imports the deleted `aaaat.security` module and no longer passes a removed `mode` argument into projection construction. Notes and right-panel editing are treated as local desktop edit operations. New/delete/reset desktop wiring was repaired. User-panel resize handling was hardened against deleted wx objects. Detailed in-place edit redraw was patched to reduce stale control artifacts after saves or cancellation.

The accepted domain model deliberately does not include the hallucinated command fields that appeared during earlier iterations. `prepare_first`, `prepare_later`, `next_action`, and `technical_reading` are not product fields. Generic inference is constrained away from lifecycle/status, material-sent state, source/lead-source invention, CV/letter state, and form answers. `source_url` or URL traceability is different from a user-facing lead-source concept. Status is `active` or `closed`.

The testing lesson from this rebuild is behavioral coverage over blacklist guards. Tests should prove the intended flows: raw-offer intake, projection shape, demo seed behavior, desktop services, bounded task application, rendering/versioning, and edit persistence. The maintainer explicitly rejected brittle tests that ban trivial words or names.

Smart View is the fast-recall/panic-mode surface for calls and low-attention review. It is read-oriented, compact, and focused on the selected candidature: offer/posting text, pitch, snapshot, fit, strategy, company/recruiter context, risks, call signals, questions, stack, evidence, keywords, and primary notes. It is not a wall of actions or a form editor. Its structure was rolled back after unstable layout churn; current files `center_cards.py`, `smart_view.py`, and `wx_html_links.py` reflect the pre-churn structured state plus the launch/security cleanup. Recent maintainer feedback strongly favored stable locations, no chaotic resizing, no right-panel overlap, no internal scrollbars inside cards, and visible text that grows instead of being silently cut.

Detailed View is the dense working surface. The accepted product shape is left candidature list/table, central complete candidature body, field-specific in-place editing, field-local generated controls near relevant fields, and a right context/options rail. The current branch has meaningful groundwork and repaint fixes, but not every v1 detailed workflow is complete.

User View is still behind the intended v1 shape. The accepted target is a proper professional/career workspace with editable user facts and context, not a pruned read-only profile card. Profile facts and exposure concepts exist in the data layer; the wx user surface still needs fuller edit coverage.

Generation and agent execution are local, provider-neutral, and bounded. Provider adapters and task-runner groundwork exist; the product is not coupled to any provider API. Generated CV, cover letter, recruiter material, form answers, company research, keywords, and strategy are candidature artifacts or detail values that should remain inspectable and editable through the desktop. Rendering is local and versioned; render failures should occur before saving broken artifacts.

Artifact/material tracking exists as groundwork rather than a finished v1 panel. The branch contains artifact lifecycle/event support and versioned output paths. The visible desktop material/task-state presentation remains incomplete.

The most important continuity lesson is that PR37 behavior is not disposable. Later work is accepted only when it preserves or completes the PR37 desktop experience instead of replacing it with browser/server concepts, broad CRUD abstractions, hidden autonomous decisions, or generic agent machinery. The release-readiness sprint starts from the merged `didacll/local-desktop-dashboard` state, with wx desktop as the canonical human runtime and the current branch history as hard-won implementation evidence.
